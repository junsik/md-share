# md-share 로드맵

상태: v1.3 완료

기준일: 2026-07-17

이 문서는 md-share가 에이전트와 메신저에서 생성된 Markdown 결과물을 안전하게 공유하는
서비스로 발전하기 위한 마일스톤과 판단 근거를 정의한다. 현재 API 계약은
[API 문서](API.md)가 정본이다.

로드맵은 날짜 대신 의존성과 완료 조건으로 관리한다. 각 마일스톤은 앞선 마일스톤을
완료한 뒤 시작하며, 범위가 바뀌면 이 문서와 해당 이슈의 비목표를 함께 갱신한다.

## 제품 경계

md-share는 Markdown 문서 한 건을 저장하고, 렌더링 링크와 원문 링크를 반환하는
채널 중립 서비스다.

- md-share는 Markdown 검증, 저장, 렌더링, 만료와 문서 수명주기를 소유한다.
- 다우오피스 같은 채널 게이트웨이는 사용자 명령 인식, 인증된 첨부 다운로드,
  첨부별 호출과 결과 회신을 소유한다.
- 게이트웨이는 Markdown 본문을 LLM에 전달하지 않고 md-share 업로드 API로 직접 전송할
  수 있어야 한다.
- 첫 파일 업로드 범위는 `.md`만 포함한다. 이미지, PDF, 문서 변환과 임의 바이너리
  보관은 포함하지 않는다.
- 공개 공유 링크는 기본적으로 읽기 전용이며, 문서 본문은 로그와 진단 이벤트에 남기지
  않는다.

모델과 작은 운영 표면에 맞는 패턴만 선택한다.

### v0.2 — Reliable Core

저장과 만료가 실패 상황에서도 일관되게 동작하도록 기반을 강화한다.

범위:

- API, 인증, Markdown 렌더링, 저장소와 만료 처리에 대한 자동화 테스트를 추가한다.
- Markdown 본문과 메타데이터를 임시 파일에 기록한 뒤 함께 게시하는 원자적 저장 절차를
  정의한다.
- 프로세스 중단으로 생긴 임시 파일과 불완전 문서를 재시작 시 안전하게 정리한다.
- 오류 로그에는 문서 본문, bearer token과 관리 token을 기록하지 않고 문서 크기,
  오류 코드와 단계처럼 진단에 필요한 정보만 남긴다.
- 저장 실패, 디스크 용량 부족과 만료 정리 실패를 구분하는 안정적인 오류 코드를 정한다.

완료 조건:

- `lint`, `typecheck`, 테스트와 production build가 CI에서 통과한다.
- 저장 단계마다 실패를 주입해도 반쪽짜리 문서가 조회되지 않는다.
- 만료된 문서와 고아 임시 파일이 재시작 후 정리되는 테스트가 통과한다.
- 렌더러의 HTML 차단, Mermaid, 코드 블록과 GFM 회귀 테스트가 존재한다.

### v0.3 — Retry-safe Upload API

게이트웨이가 네트워크 실패 후 안전하게 재시도할 수 있는 업로드 계약을 제공한다.

범위:

- `Idempotency-Key`를 지원한다. 같은 key와 같은 요청은 같은 문서를 반환하고, 같은 key에
  다른 요청이 오면 `409 Conflict`를 반환한다.
- 업로드 요청에 원본 파일명을 선택적으로 전달할 수 있게 하고, 표시 제목과 별도로
  정규화해 보관한다.
- 파일명이 전달되면 `.md` 확장자를 확인한다. MIME만 신뢰하지 않고 UTF-8 텍스트 여부,
  NUL 문자와 현재 크기 제한도 함께 검증한다.
- 재시도 여부를 판단할 수 있도록 안정적인 오류 코드와 `Retry-After` 규칙을 문서화한다.
- 동시 요청, timeout 직후 재시도와 프로세스 재시작 뒤 재시도 테스트를 추가한다.

완료 조건:

- 동일 요청을 동시에 여러 번 보내도 문서가 하나만 생성된다.
- 성공 응답을 받지 못한 호출자가 같은 key로 재시도해 원래 URL을 얻는다.
- `.md`가 아닌 파일명, binary payload, 잘못된 UTF-8과 크기 초과 요청이 저장 전에
  거절된다.
- JSON 업로드와 raw `text/markdown` 업로드의 현재 호환성이 유지된다.

### v0.4 — Channel File Sharing

사용자와 채널 게이트웨이가 Markdown 파일 자체를 공유할 수 있게 한다.

범위:

- 웹 편집기에 `.md` 파일 선택과 drag-and-drop을 추가하고, 선택한 내용을 기존 미리보기와
  공유 흐름에 연결한다.
- 채널 게이트웨이가 사용할 raw upload 예제와 timeout, 크기 제한, idempotency 규칙을
  API 문서에 함께 제공한다.
- 문서 페이지에 제목, 설명과 canonical URL을 포함한 공유용 메타데이터를 제공한다.
- 하나의 첨부 실패가 다른 첨부 처리를 막지 않도록 파일 한 건당 요청과 결과를 갖는
  채널 통합 규칙을 정의한다.
- 다우오피스 게이트웨이에서는 사용자가 명시적으로 공유를 요청한 `.md` 첨부만 다운로드해
  이 API로 전달한다. 이 어댑터는 md-share 저장소 밖에서 관리한다.

완료 조건:

- 브라우저에서 `.md` 선택, 미리보기, 업로드와 링크 열기까지의 E2E 테스트가 통과한다.
- 가짜 채널 게이트웨이로 정상 업로드, 중복 재시도, 크기 초과, timeout과 일부 첨부 실패를
  검증한다.
- 지원하지 않는 확장자는 UI와 API 양쪽에서 동일한 오류로 거절된다.
- 게이트웨이는 Markdown 본문이나 인증 정보를 채널 로그에 남기지 않는다.

### v0.5 — Lifecycle and Operations

공유 후 문서를 안전하게 관리하고 운영 상태를 관찰할 수 있게 한다.

범위:

- 생성 응답에서 한 번만 전달되는 문서별 관리 권한을 도입한다.
- 관리 권한으로 삭제와 만료 시각 변경을 수행하는 API를 제공한다.
- 본문을 노출하지 않는 저장량, 생성·만료·삭제 건수와 오류 지표를 제공한다.
- 목록 API와 운영 화면은 제목, 크기, 생성·만료 시각과 상태만 다루고 본문 미리보기를
  기본으로 제공하지 않는다.
- 삭제, 만료 sweep과 조회가 동시에 실행될 때의 동작을 명시하고 테스트한다.

완료 조건:

- 잘못된 관리 권한으로 삭제하거나 만료를 변경할 수 없다.
- 삭제와 만료 이후 렌더링·원문 URL이 일관되게 `404`를 반환한다.
- 관리 권한과 bearer token이 로그, metric label과 목록 응답에 노출되지 않는다.
- 백업과 복구 절차에서 문서와 idempotency 정보의 일관성이 검증된다.

### v1.0 — Agent-native Contract

에이전트와 외부 게이트웨이가 저장소 구현을 몰라도 통합할 수 있는 안정 계약을 확정한다.

범위:

- 업로드, 조회, 원문, 목록과 관리 API를 설명하는 OpenAPI 문서를 제공한다.
- API 호환성, 오류 코드와 지원 기간 정책을 문서화한다.
- curl, TypeScript와 Python의 최소 통합 예제를 계약 테스트로 실행한다.
- `skills/md-share/SKILL.md`, `/skill.md`, `/api.md`, `llms.txt`와 OpenAPI 사이의 핵심
  endpoint·필드 불일치를 자동 검사한다.
- 저장소 구현을 인터페이스 뒤로 분리하되, 새로운 backend는 실제 운영 요구와 복구 시험이
  확보된 경우에만 추가한다.

완료 조건:

- 외부 구현자가 OpenAPI와 agent skill만으로 업로드 후 공유 링크를 얻을 수 있다.
- 지원 예제가 CI에서 실제 서버를 대상으로 통과한다.
- 호환되지 않는 API 변경을 탐지하는 계약 테스트가 존재한다.
- 설치, 업그레이드, 백업·복구와 보안 운영 문서가 현재 동작과 일치한다.

### v1.1 — Browser-owned Documents

공유 링크를 받은 접속자와 문서를 만든 브라우저의 관리 권한을 분리한다.

범위:

- 웹에서 새 문서를 만들 때 한 번 받은 관리 token을 현재 브라우저에만 저장한다.
- `My documents`에서 이 브라우저가 관리할 수 있는 문서 목록, 공개 메타데이터 새로 고침,
  만료 변경, 영구 삭제와 로컬 권한 제거를 제공한다.
- 공유 URL에는 관리 token을 포함하지 않고 공개 문서 페이지에는 관리 UI를 제공하지 않는다.
- 브라우저 저장소 손상, 저장 불가, 권한 만료, 문서 삭제와 network 실패를 사용자에게
  구분해 알린다.
- 실행 가능한 브라우저 시나리오에서 사용자 가이드와 스크린샷을 생성하고 CI에서 동기화를
  검사한다.

완료 조건:

- 일반 공유 링크 접속자는 문서를 읽을 수 있지만 만료 변경과 삭제를 할 수 없다.
- 관리 token, Markdown 본문과 업로드 bearer token이 공유 URL이나 브라우저 목록
  메타데이터에 섞이지 않는다.
- 문서 만료 변경, 삭제, 로컬 권한 제거와 손상된 저장소 복구 단위 테스트가 통과한다.
- 별도 브라우저 context로 공개 문서 페이지에 관리 UI가 없음을 확인하는 E2E 테스트가
  통과한다.

### v1.2 — Operational Guardrails

사람의 익명 공유 흐름과 운영자 권한을 분리하고 공개 생성의 자원 사용을 제한한다.

범위:

- 익명 공유 화면에서 운영자 token 입력란을 표시하지 않는다.
- 익명 권한은 문서 생성에만 적용하고 최근 문서 목록과 저장 상태는 운영자 token으로
  보호한다.
- 운영자 token으로 관리 token을 잃은 임의 문서를 삭제할 수 있게 하되 문서 만료 변경은
  허용하지 않는다.
- 익명 업로드에 client별 제한과 process 전체 제한을 함께 적용하고 `429`, `Retry-After`,
  `X-RateLimit-*` 계약을 제공한다.
- 신뢰한 gateway의 운영자 인증 업로드는 익명 제한을 적용하지 않는다.

완료 조건:

- token 없는 웹 사용자와 API client가 문서를 만들 수 있다.
- token 없는 호출자가 최근 문서 목록과 상태를 조회할 수 없다.
- 한 문서의 manageToken과 운영자 token 권한이 테스트에서 명확히 구분된다.
- client별·process 전체 제한, 시간창 reset과 운영자 우회 테스트가 통과한다.
- 운영 배포에서 익명 UI, 401, 403, 429와 운영자 삭제를 실제 요청으로 확인한다.

### v1.3 — Administration Console

채널 게이트웨이와 결합하지 않는 운영자용 로그인과 전체 문서 관리 화면을 제공한다.

범위:

- 설치자가 운영 환경의 secret으로 관리자 ID와 8자 이상의 비밀번호를 지정한다.
- 관리자 비밀번호를 브라우저 저장소에 남기지 않고 단일 process memory의 만료되는
  HttpOnly 세션으로 교환한다.
- 로그인 시도에 client별 제한과 process 전체 제한을 함께 적용한다.
- 운영 콘솔은 저장량과 최근 문서 메타데이터만 표시하고 Markdown 본문은 가져오지 않는다.
- 관리자 세션의 문서 삭제와 로그아웃 요청은 CSRF token으로 보호한다.
- 관리자 세션은 임의 문서를 삭제할 수 있지만 만료 변경과 인증된 업로드 권한은 받지 않는다.
- headless 자동화를 위한 bearer token은 설치자가 필요할 때만 별도로 설정한다.

완료 조건:

- 이미지와 source에 관리자 ID, 비밀번호 또는 bearer token 기본값이 없다.
- 올바른 ID와 비밀번호로 로그인하고 잘못된 값과 반복 시도는 안정적인 오류로 거절된다.
- 브라우저 local storage와 API 응답에 관리자 비밀번호와 session cookie가 노출되지 않는다.
- 관리자 세션 없는 목록·상태 조회와 CSRF 없는 삭제 요청이 거절된다.
- 운영 콘솔에서 메타데이터 조회, 검색, 삭제와 로그아웃 UI 시나리오가 통과한다.
- 다우오피스 gateway를 포함한 채널별 로직이 md-share source와 계약에 추가되지 않는다.

## 의사결정 게이트

다음 기능은 마일스톤에 자동으로 포함하지 않는다. 실제 사용 사례, 위협 모델과 운영 비용을
기록한 이슈가 생기면 별도 판단한다.

| 후보 | 검토를 시작하는 조건 | 현재 판단 |
| --- | --- | --- |
| 비밀번호 보호·열람 후 삭제 | 민감 문서의 짧은 공유가 반복되고 회수 정책이 필요할 때 | v0.5 이후 수명주기 확장 후보 |
| 문서 수정·revision | 동일 URL을 유지한 정정 요구와 감사 이력 요구가 함께 확인될 때 | 불변 링크가 기본이며 공동 편집은 범위 밖 |
| object storage | 단일 volume의 용량, 고가용성 또는 복구 목표를 충족하지 못할 때 | 파일시스템을 먼저 신뢰성 있게 운영 |
| 본문 내 첨부 이미지 | 이미지 권한, 정리, 악성 콘텐츠와 외부 URL 정책을 합의했을 때 | `.md` 단독 업로드 이후 검토 |
| MCP server | HTTP API와 skill만으로 해결되지 않는 도구 탐색·인증 문제가 반복될 때 | v1.0 계약을 우선 사용 |
| 클라이언트 측 암호화 | 서버가 평문을 보지 않아야 하는 별도 위협 모델이 승인될 때 | 렌더링·검색·운영 모델이 달라지는 별도 제품 방향 |

## 비목표

- 실시간 공동 편집기와 범용 노트 서비스
- Markdown 이외 파일의 보관, 미디어 갤러리와 URL 단축 서비스
- 채널별 인증과 첨부 다운로드 구현을 md-share에 포함하는 것
- 업로드 과정에서 LLM을 사용해 문서를 변환하거나 요약하는 것
- 운영 근거 없이 저장소 backend와 배포 구성을 늘리는 것

## 마일스톤 운영 규칙

각 마일스톤 이슈에는 다음 내용을 포함한다.

1. 해결할 사용자 시나리오와 실패 시나리오
2. 이 문서에서 연결된 오픈소스 근거와 md-share에 맞게 달리 적용할 점
3. 포함 범위와 비목표
4. API·저장 형식·보안에 대한 변경
5. 자동화된 완료 조건과 복구 검증
6. README, API, agent skill과 운영 문서 동기화 항목

마일스톤 안의 기능은 각각 독립 이슈로 분리할 수 있지만, 완료 선언은 모든 완료 조건과
문서 동기화가 끝난 뒤에만 한다.
